# Pi — estudo de source

> Veredito: harness de coding **maduro e bem desenhado** (monorepo TS: `ai` multi-provider + `agent` runtime + `coding-agent` CLI + `tui`). A tese é **núcleo pequeno + API de hooks/extensões tipada onde vive todo o resto** (permissões, subagentes, plan-mode, todo, structured-output, providers à medida são *extensões*, não core). Memória de **sessão** topo-de-gama (árvore event-sourced com branch/fork/compaction), mas **não tem memória semântica durável nem recall/RAG** — confirma o fosso do mem-vector. Estudado no commit ec6311b.

## Identidade
- Agente de coding self-extensible da earendil-works, publicado como `@earendil-works/pi-coding-agent`. Monorepo de 4 pacotes: `pi-ai` (API unificada ~30+ providers), `pi-agent-core` (runtime: loop + sessões + compaction + tools), `pi-coding-agent` (CLI interativo + tools de ficheiro + extensões), `pi-tui` (UI de terminal). Linguagem: **TypeScript** (Node ≥22, sintaxe erasable/strip-only). Licença: **MIT**.
- Provider: agnóstico — fala com Anthropic, OpenAI, Google, Bedrock, Mistral, Groq, DeepSeek, xAI, Copilot, OpenRouter, etc. via `packages/ai`. Sem geração própria.

## Anatomia (como faz cada coisa)
| Termo | Como o faz (`ficheiro`) | Força/Fraqueza | vs mem-vector |
|---|---|---|---|
| regressão/autoregressão | Fora do repo: delegado aos providers em `packages/ai` (`providers/*.ts`, `api/*.ts`). | n/a | n/a — como o mem-vector, a geração é externa (CLI/API). |
| loop | Loop duplo em `agent-loop.ts`: inner processa tool-calls + **steering messages** (utilizador digita a meio, injetado antes da próxima resposta — `:167,181,253`), outer reativa em **follow-up** depois de o agente querer parar (`:257`). Tools sequenciais ou paralelas por `executionMode` da tool (`:381`). | **Força:** steering a quente + follow-up + paralelismo de tools com ordem preservada (`:502`); hooks `beforeToolCall`(bloqueia), `afterToolCall`(altera/termina), `prepareNextTurn`(troca modelo/thinking a meio — `:226`), `shouldStopAfterTurn`. | Muito mais rico que um loop simples; o steering/follow-up é exatamente o que falta a um agente-autor que recebe instruções enquanto trabalha. |
| harness | `agent-harness.ts`: **hook bus tipado** (`on`/`subscribe`), **writes de sessão diferidos** acumulados durante o turno e **flushados em save points** (`flushPendingSessionWrites :462`, `save_point :504`), máquina de fases idle/turn/compaction/branch_summary (guarda `busy`), modelo/tools/thinking **hot-swap** gravados como entradas de sessão. | **Força grande:** a disciplina "muta em buffer, persiste no save point" mantém a sessão sempre consistente (nunca corrompida a meio do turno). | Resposta direta a um achado de auditoria ao código (writes sem transação): o padrão buffer→flush-no-boundary é importável. |
| memory | **Só memória de sessão.** `session/jsonl-repo.ts` grava cada sessão em **JSONL append-only por cwd** (`encodeCwd :34`), com `fork` (`:133`). A sessão é uma **árvore event-sourced** (`session/session.ts`): entradas tipadas (message, model_change, thinking_level_change, active_tools_change, compaction, branch_summary, label, session_info…), `parentId`/`leafId`, `getPathToRoot`. | **Força:** das melhores memórias-de-sessão dos 15 (árvore navegável, fork, reconstrução por branch). **Fraqueza p/ agente-autor:** não há base de conhecimento durável. | O mem-vector precisa do oposto: conhecimento durável que **compõe** entre sessões. Pi confirma que isto é a parte a construir, não a importar. |
| recall | **Não encontrado.** Grep ao repo inteiro: zero `embedding`/`vector`/`rag`/`semantic search`/`memory_search` (único hit é um comentário "embedding the agent in other apps"). O "recall" do Pi é **replay da árvore de sessão** (`buildSessionContext`) + o resumo da compaction. | **Fraqueza** (para conhecimento) / não-objetivo (assumido). | **É o fosso do mem-vector.** Recall híbrido (FTS+vector+citações) continua por preencher em todo o campo. |
| context | `compaction/compaction.ts`: decide por **usage REAL do provider** (`calculateContextTokens`, `getLastAssistantUsage :137`), threshold `contextWindow - reserveTokens` (`:200`); **cut points preservam pares tool_use/tool_result** (`findValidCutPoints :265`); trata **split-turn** (resume o prefixo do turno à parte — `:612`); **summary incremental e estruturado** (Goal/Constraints/Progress[Done/InProgress/Blocked]/Key Decisions/Next Steps/Critical Context — `:387`); preserva ficheiros lidos/modificados através da compaction (`:593`). | **Força grande:** compaction baseada em telemetria real, invariante tool-pair, e o molde de summary é quase igual ao das tuas dailies. | Importável quase como está; o construtor de contexto do mem-vector pode adotar o cut-point e o molde estruturado. |
| tools | `coding-agent/core/tools/{bash,edit,find,grep,ls,read,write}.ts`. `write.ts` corre dentro de `withFileMutationQueue` (**lock de mutação por ficheiro**, chave = `realpath`, paralelo entre ficheiros mas série no mesmo — `file-mutation-queue.ts:32`); `WriteOperations` plugável (delegar a SSH/remoto). Args validados antes (`validateToolArguments`). | **Força:** a lock por-ficheiro é exatamente a proteção de concorrência que faltava. **Fraqueza:** o write só devolve "wrote N bytes" — **sem verificação determinística pós-escrita**. | A `withFileMutationQueue` é importável tal-e-qual para o agente-autor (escrita concorrente ao vault/DB). Mas o mem-vector precisa do verify pós-write (hermes/odysseus) que o Pi não tem. |
| system prompt/kernel | `coding-agent/core/system-prompt.ts`: injeta ficheiros de contexto do projeto em `<project_context><project_instructions path=…>` (`:62`), overrides `SYSTEM.md`(replace)/`APPEND_SYSTEM.md`(append) — **gated por trust**; secção de skills só se a tool `read` existir (`:71`). | **Força:** contexto de projeto explícito e auditável; o próprio `AGENTS.md` do repo é um ótimo exemplo de comportamento-como-regras. | Ecoa o Kernel em camadas do mem-vector (comum vs pessoal); o gate de trust sobre overrides é uma boa rede. |
| skills | `agent/harness/skills.ts`: carrega `SKILL.md` recursivamente, frontmatter `name`/`description`/`disable-model-invocation`, validação dura do nome (lowercase, igual ao dir-pai, sem `--`), honra `.gitignore`/`.ignore`; **lazy** (só name+description+location no prompt — `system-prompt.ts:15`, conteúdo lido a pedido). **Proveniência por skill** (`loadSourcedSkills :83`). | **Força:** modelo idêntico ao Claude Code + a *source* anexada a cada skill (anti skill-injection). | Valida o padrão 7 da síntese; o `source`/proveniência é o que permite quarentena de skills no mem-vector. |
| planning | **Não no core — é extensão.** `examples/extensions/plan-mode/` e `todo.ts`. | Força: planeamento opt-in sem inchar o core. | O mem-vector pode tratar tasks/plano como add-on, não como núcleo. |
| behavior | Regras-de-comportamento como ficheiro: o `AGENTS.md` do repo (curto, sem fluff, "diz se concordas antes de editar", "pergunta antes de remover"); extensão `claude-rules.ts` carrega regras estilo CLAUDE.md. | Força: comportamento versionado e legível, carregado como contexto. | Igual à secção Comportamento do mem-vector (acumula o COMO). |
| subagentes/orquestração | **Não no core — é extensão.** `examples/extensions/subagent/` e `handoff.ts`, `send-user-message.ts`. O core só expõe os hooks (BeforeAgentStart, session fork, queue) que tornam isto possível. | Força: orquestração opcional sobre um core estável. | O relay do mem-vector pode ser uma camada de extensão sobre o core, não uma reescrita do loop. |
| stop/terminação | `stopReason` `"error"`/`"aborted"` na AssistantMessage (`types.ts:25`); flag **`terminate`** por tool-result encerra o loop se TODAS as tools do batch terminam (`agent-loop.ts:544`); `shouldStopAfterTurn`. **Não há** cap de turnos/rondas/stall no core. | **Fraqueza:** sem travões anti-runaway (ao contrário de openhuman/odysseus/hermes); depende do `terminate` e da paragem natural. | O mem-vector deve manter os travões de loop (reason codes + round cap) — o Pi não os dá. |
| verificação | **Fraca no core:** sem verify determinístico pós-write. Os guard-rails são **extensões**: `confirm-destructive.ts`, `protected-paths.ts`, `dirty-repo-guard.ts`, `git-checkpoint.ts`, `permission-gate.ts`. | Força: extensível. Fraqueza: nada é imposto por omissão. | Para um agente-autor, o verify pós-escrita (ficheiro/links/task/dedupe) tem de ser do core, não opt-in. |
| permissões/sandbox | **Duas camadas.** (1) Sandbox de execução = **não-objetivo explícito**, delegado a contentores (gondolin micro-VM / Docker / OpenShell — `docs/containerization.md`, README). (2) **Project trust** (`trust-manager.ts`): ao entrar num dir com config local (`.pi/settings.json`, extensions, skills, prompts, themes, SYSTEM.md) ou `.agents/skills`, **gateia o carregamento desses recursos não-confiáveis** atrás de um prompt (Trust/Trust-parent/sessão/Não), `proper-lockfile`, decisão hierárquica (ancestral mais próximo). As permissões de tool em si são extensões (`permission-gate.ts`) via o hook `beforeToolCall`. | **Força:** o trust anti-injeção (não auto-executar config de dir não-confiável) é maduro. **Risco assumido:** zero isolamento de execução por omissão. | O trust-store hierárquico com lock é importável; o "sandbox via contentor" valida a linha de não herdar config do host às cegas (o runner isola-se). |
| providers | `packages/ai`: ~30+ providers, OAuth (`oauth.ts`), `credential-store`, adaptadores **lazy** (`api/*.lazy.ts`), provider **`faux`** para testes determinísticos; **`getApiKey` resolvido por chamada** para tokens que expiram (`agent-loop.ts:301`). | **Força:** a abstração de provider mais limpa dos 15 (lazy + faux + OAuth + per-call key). **Fraqueza:** sprawl enorme que o mem-vector não precisa. | Importar os *princípios* (per-call key, faux provider p/ testes, lazy load), não os 30 adaptadores. |

## Pontos fortes (rankeados)
1. **Arquitetura "núcleo mínimo + tudo-é-extensão" sobre um hook bus tipado** (`agent-harness.ts` `on`/`subscribe`; `extensions/types.ts` com `defineTool` + eventos Session/Context/Tool/Provider; `examples/extensions/` com ~60 extensões reais). Permissões, subagentes, plan-mode, todo, structured-output, project-trust e providers à medida são todos extensões. É o desenho mais reutilizável dos 15.
2. **Memória de sessão em árvore event-sourced** com branch/fork/`navigateTree` + auto-summary do ramo abandonado (`session.ts`, `agent-harness.ts:732`) e JSONL append-only por cwd (`jsonl-repo.ts`).
3. **Compaction por usage real + estruturada + tool-pair-safe + split-turn** com molde Goal/Progress/Decisions/Next Steps (`compaction.ts`).
4. **Lock de mutação por-ficheiro** (`file-mutation-queue.ts`) — paralelo entre ficheiros, série no mesmo, por `realpath`.
5. **Writes de sessão diferidos → flush em save points** (`agent-harness.ts:462,504`) — consistência transacional sem transação.
6. **Disciplina de provider:** `getApiKey` por chamada (tokens a expirar), provider `faux` para testes, adaptadores lazy, OAuth/credential-store (`packages/ai`).
7. **Project trust** anti-injeção, hierárquico e lockfile-safe (`trust-manager.ts`); skills com **proveniência** (`skills.ts:83`).

## O que vale importar para o mem-vector
- [ ] **O seam de extensões/hooks tipado** — modelar os módulos/add-ons do mem-vector (relay, tasks, permissões) como extensões sobre um core estável, com hooks `beforeToolCall`(bloqueia)/`afterToolCall`(verifica/termina)/`context`/`session_before_compact`. É a forma de crescer sem reescrever o loop, e encaixa no "módulos = add-ons" que já decidiste.
- [ ] **`withFileMutationQueue` quase verbatim** — lock por `realpath`, paralelo entre ficheiros e série no mesmo. Resolve diretamente a concorrência de escrita ao vault/DB.
- [ ] **Writes diferidos + flush em save point** — bufferizar mutações durante o turno e persistir nos limites (turn_end/agent_end); a sessão/estado nunca fica corrompida a meio.
- [ ] **Compaction por usage real do provider, tool-pair-safe, com molde estruturado** — adotar o cut-point que nunca parte pares tool_use/tool_result e o summary Goal/Progress/Decisions/Next Steps (é quase a tua daily). Reaproveita o `getLastAssistantUsage`.
- [ ] **`getApiKey` por chamada + provider `faux` para testes** — barato, e o faux destrava testes do loop sem gastar tokens (encaixa no relay Claude↔Codex).
- [ ] **Project-trust hierárquico com lockfile** — não auto-carregar config/skills/extensões de fonte não-confiável sem decisão explícita; reforça a quarentena de skills (padrão 7) e o princípio de não herdar config do host.
- [ ] **Steering + follow-up no loop** — deixar o utilizador injetar instruções enquanto o agente-autor trabalha, processadas antes do próximo passo (`getSteeringMessages`/`getFollowUpMessages`).

## Não importar / armadilhas
- **Não importar os ~30 providers nem a stack TUI/RPC inteira** — o mem-vector usa Claude+Codex; ficar com os *princípios* de provider (per-call key, faux, lazy), não os adaptadores.
- **Não copiar o modelo de permissões do Pi como base de segurança** — o Pi assume **zero sandbox de execução** por omissão (delega a contentores). Para um agente-autor que escreve sozinho, o gate destrutivo e o verify pós-write têm de ser do **core**, não extensões opt-in.
- **Não herdar a ausência de travões de loop** — o Pi não tem cap de turnos/rondas/stall; manter os reason codes + round cap do mem-vector.
- **Não confundir a memória de sessão (excelente) com memória de conhecimento (inexistente)** — a árvore JSONL é para conversas, não é o vault que compõe. O recall/RAG continua a ser a parte a construir.
- **`navigateTree`/fork é poderoso mas é estado de UI de sessão** — não o transformar no modelo de versões do conhecimento sem pensar (o mem-vector já tem `file_versions`/edges).
