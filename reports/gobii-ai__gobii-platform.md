# Gobii Platform — estudo de source

> Veredito: é uma plataforma real de agentes persistentes, não uma demo: junta loop Celery/Redis, memória relacional, browser tasks, tools, sandbox, compaction e evals. Para o mem-vector, a maior importação é a disciplina entre estado efémero, memória durável e contexto compactado, não a stack inteira. Estudado no commit 865028a.

## Identidade
- O que é: plataforma Django para agentes persistentes e browser agents, com `PersistentAgent` agendado/recorrente e `BrowserUseAgentTask` para tarefas browser (`clones/gobii-ai__gobii-platform/api/models.py:2520`, `clones/gobii-ai__gobii-platform/api/models.py:6627`).
- Provider, linguagem, licença: providers via LiteLLM e endpoints configurados em DB, com backends OpenAI/Anthropic/Google/OpenAI-compatible no schema (`clones/gobii-ai__gobii-platform/api/models.py:3280`, `clones/gobii-ai__gobii-platform/api/models.py:3336`); backend Python/Django/Celery (`clones/gobii-ai__gobii-platform/pyproject.toml:1`), frontend React/TypeScript (`clones/gobii-ai__gobii-platform/frontend/package.json:1`); licença MIT (`clones/gobii-ai__gobii-platform/LICENSE:1`).

## Pontos fortes (rankeados)
1. Compaction incremental de comunicações e steps com snapshots, cauda raw e trabalho LLM fora do lock (`clones/gobii-ai__gobii-platform/api/agent/core/compaction.py:97`, `clones/gobii-ai__gobii-platform/api/agent/core/step_compaction.py:148`).
2. SQLite como estado de trabalho editável pelo agente, com tabelas builtin para config, kanban, mensagens, ficheiros, contactos e skills (`clones/gobii-ai__gobii-platform/api/agent/tools/sqlite_state.py:32`).
3. Prompt budget tree com shrinker head/mid/tail, útil para preservar forma do contexto sob limite de tokens (`clones/gobii-ai__gobii-platform/api/agent/core/promptree.py:16`, `clones/gobii-ai__gobii-platform/api/agent/core/promptree.py:173`).
4. Evals comportamentais para distinguir memória durável de preferências one-off (`clones/gobii-ai__gobii-platform/api/evals/scenarios/behavior_micro.py:1774`, `clones/gobii-ai__gobii-platform/api/evals/scenarios/behavior_micro.py:1905`).
5. Tool registry com caps, LRU e tools obrigatórias por skill (`clones/gobii-ai__gobii-platform/api/agent/tools/tool_manager.py:486`, `clones/gobii-ai__gobii-platform/api/agent/tools/tool_manager.py:604`).
6. Auditoria rica: completions, steps, tool calls, system steps e prompt archive (`clones/gobii-ai__gobii-platform/api/models.py:12249`, `clones/gobii-ai__gobii-platform/api/models.py:12440`, `clones/gobii-ai__gobii-platform/api/models.py:12791`).
7. Modelo de relay agente-a-agente com quotas e limites de autoridade (`clones/gobii-ai__gobii-platform/api/models.py:11812`, `clones/gobii-ai__gobii-platform/api/agent/core/prompt_context.py:3776`).

## O que vale importar para o mem-vector
- [ ] SQLite working-state + reconciliação para vault/DB — usar uma DB efémera por execução para `tasks`, `daily`, `contacts`, `tool_results`, `agent_config` e `skills`, aplicando só diffs validados ao vault durável; inspiração directa em `sqlite_state.py`, `sqlite_agent_config.py` e `sqlite_skills.py`.
- [ ] Compaction dual: chat e steps — manter snapshots incrementais separados para conversas e acções, com cauda raw recente e sem apagar eventos originais; encaixa na camada de memória longa do mem-vector.
- [ ] Promptree/token budget — representar contexto como árvore com prioridades e shrinkers, para equilibrar charter, RAG, daily, tasks, relay e histórico sem concatenar tudo.
- [ ] Evals de memória durável vs efémera — criar cenários tipo “guardar preferência”, “não guardar preferência one-off”, “actualizar regra existente” antes de deixar o agente escrever no vault.
- [ ] Skills como playbooks versionados — guardar workflows recorrentes no vault/DB e expô-los no contexto como memória operacional, não como código hardcoded.
- [ ] Relay Claude<->Codex com peer model — adaptar `AgentPeerLink`/quotas/authority para impedir que um peer reescreva charter, permissões ou memória durável sem canal autorizado.
- [ ] Audit log completo — persistir completion, prompt archive, tool calls, diffs de DB e system steps para replay/debug.
- [ ] Termination budget explícito — limites de passos, streak sem tools, sleep/resume e razão de paragem em cada ciclo.
- [ ] Sandbox separado para shell/Python — útil para tarefas de coding, mas com política mais apertada para paths do vault e sem assumir que `shell=True` é seguro fora de contentor.

## Não importar / armadilhas
- Não importar a stack inteira Django/Celery/Redis/billing/waffle: é produto multi-tenant, demasiado pesado para um agente-autor pessoal.
- Não copiar “charter + prompt gigante” como substituto de memória: vector/RAG geral não foi encontrado; mem-vector deve manter retrieval semântico próprio.
- Não tratar SQLite efémero como fonte de verdade; o padrão bom é reconciliar mudanças para a memória durável.
- Não deixar o agente escrever memória durável sem evals/guards de transient-vs-durable; o próprio repo precisa de cenários específicos para isto.
- Não importar implied-send semantics se o relay Claude<->Codex precisa de entregas explícitas e auditáveis.
- Não copiar provider routing/billing antes de precisar; uma interface provider mínima com fallback chega.
- Não confiar só no LLM judge para verificação; usar verificadores determinísticos para integridade do vault, links, tasks e summaries.
- Não abrir sandbox a paths reais do vault por defeito; a normalização de workspace ajuda, mas comandos shell continuam perigosos.
